# 方案分阶段与理由（通俗易懂）

## 阶段 A｜零侵入配置（立刻可做，收益快）

1. 调参基线
   - k=8，searchType=mmr，mmrLambda=0.35
   - 理由：MMR 让结果“有代表性不重复”；k 稍大便于后续重排精选。
2. 上下文长度
   - RAG_CTX_CHAR_LIMIT=3500~4500
   - 理由：保证能放下3–5段高质量证据与来源，不至于被模型截断。
3. 工具优先级
   - tools 顺序：kb_search 放前，tavily 在后
   - 理由：先用内部知识减少“张冠李戴”，外部搜索作兜底。
4. 强制“要有引用” 
   - 在系统/开发指令中明确“回答必须带来源序号[1][2]，缺证据要说明” 
   - 理由：引导模型复述检索内容而非“编”。

## 阶段 B｜中等改动（需要少量开发，准确率大幅提升）

1. 多查询扩展（Multi-Query）
- 将用户问题改写出 3–5 个等价/近义查询，合并召回后再重排
- 理由：一个问法覆盖面有限，多问几种说法能找全、找准。
2. Cross-Encoder 重排
- 先向量召回 Top-20，再用重排器选 Top-3~5 入上下文
- 理由：向量相似度只看“像不像”，重排看“有没有直接回答问题”，更接近最终需求。
3. 元数据过滤
- 按模块/版本/语言/更新时间等字段在检索前过滤（where 预过滤）
- 理由：先缩小搜索面，再做语义检索，精准度显著提升。
4. 结构化上下文拼接
- 每段保留：标题/小节/版本/更新时间/来源；顶部给2行“检索摘要”
- 理由：模型看到“证据卡片”更容易引用且不跑题。
5. 去重与同源抑制
- 同一来源相近段只留分数最高者；标题相同仅保留1条
- 理由：信息不重复=更省 token、上下文更干净。
## 阶段 C｜高级策略（稳定性与可运维性）

1. 任务路由
- 内部知识类→优先 kb_search；开放事实/新闻类→优先外部搜索；拿不准→并行召回后统一重排
- 理由：把“问谁更懂”这件事交给路由器，减少无谓尝试。
2. 低置信度二跳
- 如果命中分低/引用不足→自动二次扩检（更大 k、改写查询）再重排
- 理由：不给“糊弄答案”，先把证据找齐。
3. 可观测与评测
- 落日志：查询→候选→重排→最终段→引用→响应时延；线下评测 nDCG/Hit@K/含引文率/幻觉率
- 理由：让优化“可量化可回溯”，迭代不走盲路。
4. 权限与多租户
- 在检索层做 ACL 过滤（tenantId/role），不同用户只见到有权限的文档
- 理由：数据安全与合规的底线。

### 落地优先级与预期效果

- 优先级：A（当天完成）→ B（1–3 天）→ C（按需推进）
预期（经验值，供参考）
- A：含引文率↑、幻觉率↓，命中率提升 10%+
- B：Top-1 相关性明显提升，误召误答显著降低（常见可到 20–40%）
- C：稳定性、可维护性显著提高

## 方案A运行指导（落地操作）

- 环境基线
  - `CHROMA_URL` 指向可用的 Chroma 服务（如 `http://localhost:8000`）。
  - `KB_COLLECTION` 指向目标集合名（确保已入库）。
  - `KB_STORAGE_ROOT` 为绝对路径；`KB_STORAGE_RAW_DIR` 为空时自动回退到 `<KB_STORAGE_ROOT>/raw`。
  - `KB_EMBED_PROVIDER` 与 `KB_EMBED_MODEL` 按语种选择（示例：`gemini + gemini-embedding-001` 或 `openai + text-embedding-3-small`）。
  - `RAG_CTX_CHAR_LIMIT=4000`（若上下文易被截断，可在 3500–4500 间微调）。

- 推荐调用参数（通过提示/习惯用法引导模型）
  - 常规：`searchType=mmr`，`mmrLambda=0.35`，`k=8`。
  - 窄域/明确定位问题：`searchType=similarity`，`k=6`。

- 提示语模板（使模型更易选择正确工具与参数）
  - 内部事实问题：
    - “请先用内部检索工具 kb_search，使用 MMR（mmrLambda=0.35，k=8）检索，再基于检索结果回答，并在答案中引用来源编号。”
  - 版本/变更类问题：
    - “优先检索与模块 X 相关的文档，若找到多个版本，以最新更新时间为准，引用来源并标注版本号。”

- 答案规范
  - 必须附带引用（如 `[1][2]`），并与上下文“参考来源”编号对齐。
  - 当最高相关度较低/证据不足时，应明确说明并建议进一步检索，而非臆测作答。

- 观测与抽样核对
  - CLI 运行时关注日志中的 `[tool-call]` 与 `[tool-result]`，检查是否优先触发 `kb_search`、检索类型是否为 `mmr`、`k` 是否在 6–8。
  - 抽样 10 次问答，记录：是否触发检索、是否引用来源、端到端时延变化。目标：检索触发率 ≥90%，含引文率 ≥90%，P50 增幅 ≤300ms。

## “MMR 最小改动”使用指引（方案B）

- 适用目标
  - 让 MMR 在 Chroma 集成下稳定生效，通过放大候选集（fetchK）获得更强的多样性与覆盖。

- 工具参数（kb_search）
  - `searchType`: `mmr | similarity`（推荐 `mmr`）
  - `mmrLambda`: 0.35（可调 0.3–0.5）
  - `k`: 8（可按 token 预算 6–10）
  - `fetchK`: 32（或 `4*k`）

- 示例调用（提示语）
  - “请先用 kb_search，searchType=mmr, mmrLambda=0.35, fetchK=32, k=8 检索，再结合检索结果回答，并在答案中引用来源编号。”

- 注意事项
  - 当后端/驱动暂不支持 MMR 时，模型可能自动补发一次 `similarity` 检索作为回退；这是期望内行为。
  - fetchK 增大会增加检索时间，建议从 32 起步，观察 P50 时延再调优。
  - 若上下文“参考来源”为空，通常是入库时未写入 `metadata.source`，需在入库链路补齐（进入方案B后续事项）。
